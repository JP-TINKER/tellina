# Tellina v2.1 plan: RAM-first intake (opt-in prototype)

An optional mode where the browser writes downloads straight into a
memory-backed folder that Tellina watches, instead of onto the disk. Tellina
scans the file where it lands, promotes it to `~/Downloads` only if it's
clean, and deletes anything malicious without it ever touching the disk.

**Plan only, no code yet.** Nothing changes by default; this is a strict
opt-in layered on top of v2's normal behavior.

## Why bother

v2 already reacts fast, but the browser still writes the file to disk first,
and Tellina acts after the fact. This mode fixes three things that follow
from that:

- **An unscanned file never touches the disk.** It's born in memory; if it
  turns out to be malware, it's deleted from memory and a disk copy never
  existed.
- **One disk write instead of two.** v2 writes a clean small file to disk
  twice (the download, then the release copy). This mode writes it once,
  same as an unguarded download.
- **No window where an unscanned file sits in `~/Downloads`.** Nothing shows
  up there until it's already passed.

Longer term, because nothing is released until the verdict is in, scan time
stops costing exposure time. That opens the door to slower, more thorough
checks later without leaving the user exposed while they run. This plan
doesn't add any such checks; it just makes room for them.

## What this is not

Detection doesn't change. Same ClamAV signatures, same disguised-executable
and desktop-launcher checks, same blind spots. This is about *where the file
sits* while it's being checked, not about catching more malware. (A separate
idea about detecting tracking beacons in documents was raised alongside this
one and was deliberately kept out of Tellina; that belongs in a different
tool.)

## Staying inside the rules

The v2 hard rules hold, with one honest stretch. Rules 1 through 5, no new
packages, no root, no network calls, fail-open, silent on pass, are all
unchanged.

The stretch is rule 6, no settings screen and no runtime toggle. This mode is
switched on once, at setup, via an environment variable and a setup script,
the same way v2 already exposes a few operator settings. There's still
nothing to click while the service runs. The real setup cost is a one-time
browser setting (see below), which lives outside Tellina entirely.

## How it would work

- `DOWNLOAD_RAM_FIRST=1` turns it on. A helper script creates the intake
  folder and flips this on for the service.
- The intake folder defaults to `/dev/shm/tellina-$UID/intake` (`/dev/shm` is
  usually about half your RAM, versus roughly a tenth for the runtime dir
  v2 uses, so it fits more real downloads).
- You point your browser's download folder at that intake folder, once. A
  helper could automate this for some browsers, but browser settings change
  across versions, so a documented manual step is the honest default.
- Tellina watches the intake folder instead of `~/Downloads`. A file lands,
  gets checked exactly like a held file today (launcher check, disguise
  check, structure check, ClamAV scan), and then:
  - **Clean**: copied to `~/Downloads` (one disk write), removed from intake.
  - **Malicious**: deleted from intake. Nothing reaches disk.
  - **Unknown** (corrupt, tampered): moved to quarantine, same as today, so a
    mistake stays recoverable, at the cost of that one file touching disk.
  - **Too large**: released unscanned rather than trapped somewhere the user
    can't reach it, logged clearly either way.
- Nearly everything else, the check pipeline, quarantine, crash recovery,
  carries over unchanged. The one part that goes away is the move-to-hold
  step, since the file already starts in memory.

## Honest limits

- **Size is capped by RAM, and there's no graceful fallback.** The browser
  commits to one destination folder before it knows how big a file will be,
  so a download bigger than the intake budget just fails, it doesn't spill
  over to disk. `/dev/shm` covers most everyday downloads, not a large ISO.
  There's no way to fix this without a new dependency (see rejected, below).
- **"Never touches disk" isn't airtight.** Under real memory pressure, the
  kernel can page memory-backed files to swap, so bytes could transiently
  reach disk unless swap is off or encrypted.
- **It's volatile.** A crash, logout, or power cut wipes the intake folder,
  losing anything mid-download or not yet promoted.
- **It only covers browser downloads aimed at that folder.** A "Save As"
  elsewhere, a torrent client, or `curl` bypasses it entirely.
- **No detection gain.** Worth repeating: same signatures, same heuristics.

## Open questions for the prototype

- Should Tellina also keep watching `~/Downloads` as a backstop for tools
  that don't use the intake folder?
- Is the fingerprint check still worth doing when a file only ever sits in
  the user's own memory folder?
- How far should the setup helper go in automating the browser setting
  before it's more fragile than it's worth?

## Rejected alternatives

- **A FUSE overlay** that keeps small files in memory and spills large ones
  to disk automatically. This is the only way to get true size-based
  fallback without the browser knowing, but it needs a compiled dependency
  (`libfuse`), which breaks rule 1.
- **A browser extension** to intercept downloads directly. More complete,
  but a new dependency per browser, with store publishing and constant
  breakage across updates.
- **A local scanning proxy**, the way network appliances do inline scanning.
  Needs a proxy and HTTPS interception with a local certificate: a big
  dependency and a real trust change.
- **Resizing or mounting a custom memory filesystem** to escape the RAM size
  cap. Needs root.

## Rollout plan

Ships off. Fully reversible: turn off the flag, point the browser back at
`~/Downloads`, and nothing is left behind. Success looks like: a clean
download appears in `~/Downloads` with exactly one disk write and never shows
up there before it's cleared; a malicious one leaves no disk trace at all; an
oversize one either fits in memory or fails loudly, never slips through
unscanned.
